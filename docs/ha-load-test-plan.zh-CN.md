## 负载测试计划

### 测试场景

#### 1. 注册突发测试
- **目标**：2000 个实例在 2 分钟内注册
- **测试内容**：验证系统在突发注册压力下的稳定性
- **关注指标**：注册接口延迟、错误率、数据库写入性能

#### 2. 心跳稳态测试
- **目标**：5000 个实例，每 15 秒发送一次心跳
- **测试内容**：模拟生产环境的持续心跳负载
- **关注指标**：心跳接口延迟、CPU/内存使用率、数据库更新性能

#### 3. 服务发现读取测试
- **目标**：200 RPS（每秒请求数）的列表/订阅请求，使用 ETag
- **测试内容**：验证服务发现的读取性能和缓存效果
- **关注指标**：查询延迟、长轮询连接数、唤醒延迟

#### 4. 配置发布测试
- **目标**：50 RPS 的配置更新，包含历史记录写入
- **测试内容**：验证配置管理的写入性能和版本管理
- **关注指标**：更新延迟、历史记录写入性能、并发更新冲突率

### 收集的指标

#### 性能指标
- **延迟**：
  - P50（中位数）延迟
  - P95（第 95 百分位）延迟
  - P99（第 99 百分位）延迟
- **吞吐量**：每秒请求数（RPS）
- **错误率**：4xx/5xx 响应占比

#### 资源指标
- **API 节点**：CPU 使用率、内存使用率、网络 I/O
- **数据库**：CPU 使用率、内存使用率、磁盘 I/O、连接数
- **长轮询**：活跃连接数、唤醒延迟、超时率

#### 业务指标
- **注册中心**：注册成功率、心跳丢失率、实例健康状态变化
- **配置管理**：更新成功率、版本冲突率、订阅通知延迟
- **审计日志**：日志完整性、写入延迟

### 测试工具

#### 负载生成工具
- **k6**：推荐，内置 JavaScript 脚本，丰富的指标输出
- **Vegeta**：轻量级 HTTP 负载工具，命令行友好
- **JMeter**：功能全面，适合复杂场景

#### 环境部署
- **Docker Compose**：本地基线测试
- **Kubernetes**：生产环境模拟测试

#### 监控工具
- **Prometheus**：指标收集和存储
- **Grafana**：可视化仪表板
- **OpenTelemetry**：分布式追踪和指标

### 测试脚本示例（k6）

```javascript
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '30s', target: 100 },  // 30秒内增加到100并发
    { duration: '2m', target: 100 },   // 保持2分钟
    { duration: '30s', target: 0 },    // 30秒内降至0
  ],
};

export default function () {
  // 注册实例
  const registerPayload = JSON.stringify({
    namespace: 'default',
    group: 'DEFAULT_GROUP',
    serviceName: 'test-service',
    host: '127.0.0.1',
    port: Math.floor(Math.random() * 10000) + 10000,
    weight: 100,
    ttlSeconds: 30,
    isEphemeral: true,
  });

  const registerRes = http.post('http://localhost:5043/api/v1/registry/register', 
    registerPayload, 
    { headers: { 'Content-Type': 'application/json' } }
  );
  
  check(registerRes, {
    'register status is 200': (r) => r.status === 200,
    'register latency < 200ms': (r) => r.timings.duration < 200,
  });

  sleep(1);
}
```

## 通过标准

### 性能要求
- ✅ 读取端点 P95 延迟低于 200ms
- ✅ 写入端点 P95 延迟低于 500ms
- ✅ 长轮询唤醒延迟低于 100ms

### 稳定性要求
- ✅ 持续负载下错误率低于 0.5%
- ✅ 无内存泄漏或资源泄漏
- ✅ 数据库连接池无耗尽

### 数据完整性
- ✅ 配置历史记录无丢失
- ✅ 审计日志完整记录所有操作
- ✅ 实例状态变更及时反映

## 后续优化计划

### 1. 连接池调优
- 根据实际负载调整数据库连接池大小
- 配置连接空闲超时和最大生命周期
- 监控连接池使用率和等待时间

### 2. 缓存优化
- 为服务发现读取添加内存缓存
- 实现分布式缓存（Redis）用于跨节点共享
- 配置合理的缓存过期策略

### 3. 数据分区策略
- 配置历史记录按时间分区
- 审计日志按月份分区归档
- 考虑冷热数据分离

### 4. 异步处理
- 审计日志异步写入（队列化）
- 配置历史清理异步执行
- 通知推送采用后台任务

### 5. 监控告警
- 设置性能下降告警阈值
- 配置错误率突增告警
- 监控数据库慢查询

### 6. 压力测试扩展
- 测试更大规模（10k+ 实例）
- 模拟网络分区和节点故障
- 测试长时间运行的稳定性（Soak Test）

## 生产环境部署建议

### 资源配置
- **API 节点**：4 vCPU、8GB RAM（每节点）
- **数据库**：8 vCPU、16GB RAM、SSD 存储
- **负载均衡器**：支持 TCP/HTTP 健康检查

### 配额设置
```json
{
  "Quota": {
    "MaxServicesPerNamespace": 2000,
    "MaxInstancesPerNamespace": 10000,
    "MaxConfigsPerNamespace": 1000,
    "MaxConfigContentLength": 1048576
  }
}
```

### 健康检查配置
```json
{
  "Registry": {
    "Health": {
      "CheckIntervalSeconds": 10,
      "MarkUnhealthyAfterSeconds": 60
    }
  }
}
```

### 备份策略
- **数据库备份**：每日全量备份 + 实时 binlog
- **配置备份**：定期导出配置快照
- **审计日志归档**：按月归档至对象存储

## 故障演练

### 1. 单节点故障
- **场景**：模拟单个 API 节点崩溃
- **预期**：负载均衡器自动摘除故障节点，流量转移到健康节点
- **验证**：服务无中断，延迟在可接受范围内

### 2. 数据库主从切换
- **场景**：主数据库故障，切换到从库
- **预期**：短暂服务中断（< 30 秒），数据无丢失
- **验证**：切换后服务恢复正常，数据一致性正确

### 3. 网络分区
- **场景**：部分节点与数据库断开连接
- **预期**：受影响节点返回错误，其他节点正常服务
- **验证**：网络恢复后自动恢复，无需人工干预

## 结论

通过系统的负载测试和高可用部署，SiNan 平台可以满足大规模生产环境的性能和可用性要求。持续的监控、优化和故障演练是保持系统稳定运行的关键。
